# air-harness

Личный харнес для работы с Claude Code: процесс, роли агентов, ритуалы и
гейты качества, переиспользуемые между всеми проектами.

Ставится как плагин Claude Code — один раз в каждый проект. Правится в одном
месте, обновляется везде.

## Зачем

Каждый новый проект (репозиторий, микросервис, прототип) требовал одного и
того же: настроить процесс работы с агентами, завести доску задач, описать
соглашения, расставить гейты. Раз за разом вручную, каждый раз чуть иначе.

Харнес делает это один раз и навсегда:

- **Процесс** — задача → ветка → worktree → PR → ревью → мерж → уборка.
- **Роли** — исполнитель, ревьюер, декомпозитор, документатор.
- **Ритуалы** — `/board`, `/groom`, `/task`, `/land` вместо абзацев промпта.
- **Гейты** — хуки, которые не дают нарушить процесс, даже если агент забыл.
- **Инициализация** — `/new-project` разворачивает всё это в новом репозитории
  за один заход.

## Установка

```bash
/plugin marketplace add ErnsetAirapetov/air-harness
/plugin install harness-core@air-harness
```

Для git-хоста, отличного от GitHub (например корпоративный self-hosted GitLab),
источником служит полный URL:

```bash
/plugin marketplace add https://git.corp.local/tools/air-harness.git
```

### Автообновление — включить обязательно

Для сторонних маркетплейсов (этот — сторонний) автообновление **по умолчанию
выключено**: установленная копия плагина кэшируется и сама не обновляется.
Без этого главное свойство харнеса — «починил в одном месте, работает везде» —
не работает. Один раз на каждой машине:

```
/plugin → вкладка Marketplaces → air-harness → Auto-update: on
```

После этого Claude Code проверяет обновления после старта сессии и подтягивает
свежий `main`. Обновить вручную: `/plugin marketplace update air-harness`.

### Если что-то не работает

Два случая, которые легко перепутать:

- **Агентов и скилов харнеса не видно** (`/agents`, `/plugin` их не
  показывают) — плагин не загрузился; разбирайся с установкой и маркетплейсом.
- **Агенты и скилы на месте, а хартии оркестратора в контексте нет** — это не
  сбой плагина. Хартию вкладывает SessionStart-хук, а он не стреляет при
  подключении плагина посреди сессии (`/reload-plugins`) — только на старте,
  `/resume`, `/clear` и `/compact`. Лечится `/compact` или перезапуском
  сессии; тот же текст можно впрыснуть вручную:
  `node <корень плагина>/scripts/orchestrator-context.mjs`.

## Использование

В новом пустом репозитории:

```
/new-project
```

Команда опросит про стек, фордж и назначение проекта, развернёт локальную
обвязку, заведёт метки на доске и создаст первые задачи.

В существующем репозитории, где уже есть код:

```
/harness-init
```

Дальше — повседневный цикл:

```
/board            что происходит, что ждёт моего решения
/groom <цель>     разложить цель на атомарные задачи
/task 12 13 14    взять задачи в работу, параллельно
/land 7           отревьюить, замержить, убрать за собой
```

## Каталог возможностей

Полный состав того, что плагин добавляет в проект. Каталог обязан совпадать с
фактическим содержимым `core/` — при изменении состава обновляется тем же PR.
Подробности живут в самих файлах команд, скилов, агентов и скриптов; здесь —
только «что и когда».

### Ритуалы владельца

Владелец вызывает сам, командой прямо в сессии оркестратора:

- **`/board`** — сводка доски для того, кто не читает код: что ждёт решения, что
  в работе, что готово к мержу, аномалии. Зови, когда нужна картина текущего
  состояния.
- **`/groom <цель>`** — раскладывает цель или эпик на атомарные задачи доски
  (через агента `groomer`, план показывается до создания). Зови в начале нового
  направления.
- **`/task <номера>`** — берёт задачи в работу: worktree, агент-исполнитель, PR;
  несколько номеров — параллельно. Зови, когда задача готова к реализации.
- **`/land <номера PR>`** — ревью PR агентом `reviewer` и, если чисто, мерж с
  полной уборкой (worktree, ветка, закрытие задачи). Зови, когда PR готов.
- **`/handoff`** — сохраняет состояние сессии в `.claude/handoff.md` перед
  `/compact` или паузой. Зови, когда контекст подходит к пределу.
- **`/handoff limit`** — тот же хендофф плюс команда планировщика ОС на момент
  сброса 5-часового лимита. Зови, когда работу прервал лимит.
- **`/harness-init`** — разворачивает харнес в репозитории: привязки
  (`harness.json`, `forge.md`, `settings.json`), метки и шаблон задачи на доске,
  защита главной ветки на фордже. Зови один раз при подключении проекта.
- **`/harness-feedback`** — заводит issue об ошибке самого харнеса (сбойный хук,
  скил, агент, пробел протокола) в доме харнеса по протоколу. Зови, когда сбой в
  обвязке, а не в боевом проекте.

### Агенты

Роли-исполнители. Их спаунит оркестратор через ритуалы — владелец напрямую их
не зовёт:

- **`impl`** (opus) — исполнитель одной задачи: свой worktree, код с проверками,
  коммит, PR. Спаунится из `/task` (и из `/land` — на доработку).
- **`reviewer`** (opus, только чтение) — ревьюер PR: критерии приёмки, границы,
  соглашения, реальные дефекты. Спаунится из `/land`.
- **`groomer`** (sonnet) — декомпозирует эпик в атомарные задачи с метками и
  зависимостями; файлы репозитория не правит, пишет только на доску. Спаунится
  из `/groom`.
- **`scaffolder`** (sonnet) — разворачивает стартовую документную структуру
  (`CLAUDE.md`, `AGENTS.md`, `docs/`) и возвращает вопросы владельцу. Спаунится
  при инициализации проекта.

### Гейты (хуки)

Код, который держит процесс, даже если агент забыл. Подключаются автоматически
из `core/hooks/hooks.json`:

- **guard-git** (PreToolUse на Bash) — не даёт нарушить git-процесс: прямой
  коммит в главную ветку, чужое авторство, упоминания ассистентов. Разбирает
  команду, а не грепает сырую строку.
- **edit-checks** (PostToolUse на Write\|Edit) — мгновенный фидбек после правки
  файла по правилам `editChecks` из `harness.json`, чтобы ошибки чинились сразу.
- **run-checks** (Stop и SubagentStop) — не даёт закончить молча с красными
  проверками: гоняет команды `checks` из `harness.json`.
- **orchestrator-context** (SessionStart) — вкладывает в контекст главной сессии
  хартию оркестратора: единственный канал правил, раз плагин не поставляет
  `CLAUDE.md`.

## Обратная связь

Канал фидбэка об ошибках самого харнеса — issues этого репозитория. Агентам в
подключённых проектах протокол задаёт скилл `harness-feedback`: где заводить,
как проверить дубликаты, что писать и чего в тексте быть не должно.
