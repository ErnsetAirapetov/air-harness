# Архитектура

## Задача

Один и тот же процесс работы с роем агентов нужно применять в разных
репозиториях, на разных стеках, с разными досками задач — и уметь улучшать его
в одном месте.

Наивное решение — копировать `.claude/` из проекта в проект — разваливается на
третьем репозитории: копии расходятся, исправление приходится вносить N раз,
и никто уже не помнит, какая версия правильная. Это и есть исходная проблема.

## Решение: три слоя

```
┌─────────────────────────────────────────────────────────┐
│  ~/.claude/          ЛИЧНЫЙ слой                        │
│  Что верно для владельца всегда и везде:                │
│  стиль коммитов, запрет чужого авторства, «упёрся —     │
│  спроси, не выдумывай»                                  │
├─────────────────────────────────────────────────────────┤
│  плагин harness-core   ПРОЦЕССНЫЙ слой  ← этот репозиторий │
│  Роли агентов, ритуалы, гейты, процедура инициализации.  │
│  Не знает ни про фордж, ни про стек, ни про домен.       │
├─────────────────────────────────────────────────────────┤
│  .claude/ проекта     СЛОЙ ПРИВЯЗОК                     │
│  harness.json, forge.md, settings.json — генерируются    │
│  init-скиллом. Единственное место с конкретикой.         │
├─────────────────────────────────────────────────────────┤
│  CLAUDE.md проекта    ПРЕДМЕТНЫЙ слой                   │
│  Домен, модули, соглашения конкретного проекта           │
└─────────────────────────────────────────────────────────┘
```

Слои отличаются **частотой изменения**. Личный слой почти не меняется.
Процессный — от накопленного опыта, раз в недели. Привязки — при смене форджа
или стека. Предметный — постоянно. Смешивать слои с разной частотой изменения
— главный источник боли, ради устранения которой всё и затевалось.

## Точки перевода

Изменчивость не размазывается по файлам, а собирается в одну точку на каждую ось.

### Фордж — `.claude/forge.md`

Агент никогда не пишет `gh pr create`. Агент пишет «создай PR», а *как именно* —
читает из таблицы в проекте:

```markdown
Фордж: GitHub (gh CLI)

прочитать задачу   gh issue view <N>
создать задачу     gh issue create --title "<t>" --body-file <f> --label <l>
создать PR         gh pr create --base main --title "<t>" --body "<b>"
смержить PR        gh pr merge <N> --squash --delete-branch
пометить решением  gh issue edit <N> --add-label needs-decision
закрыть задачу     gh issue close <N> -c "<итог>"
```

Для GitLab та же таблица с `glab mr create --target-branch main`. Для Plane —
с `curl` к его API. **Агенты, команды и хуки при этом не меняются вообще.**

Переезд между досками задач — это переписать одну таблицу, а не вычитать
восемь файлов. Проверено дорогой ценой: до этого переезд с Plane на GitHub
Issues означал переделку всего.

### Стек и владелец — `.claude/harness.json`

Машиночитаемое, для скриптов хуков:

```json
{
  "mainBranch": "main",
  "ownerEmail": "eriktarakan@gmail.com",
  "worktreeDir": ".claude/worktrees",
  "checks": ["npm test", "npx tsc --noEmit"]
}
```

Хук `run-checks.mjs` не знает про npm — он берёт список команд отсюда. Хук
`guard-git.mjs` не знает про `main` — берёт имя ветки отсюда.

## Компоненты плагина и их роли

| Компонент | Назначение | Кто читает |
|---|---|---|
| `agents/` | Роли. Стабильный маленький набор: исполнитель, ревьюер, декомпозитор, документатор | оркестратор при делегировании |
| `commands/` | Ритуалы человека. Одна строка вместо абзаца промпта | владелец |
| `skills/` | Процедуры, которые модель применяет сама или по вызову. В том числе инициализация проекта | модель |
| `hooks/` + `scripts/` | Гейты. Выполняются харнессом, а не моделью — их нельзя «забыть» | харнесс |

### Почему гейты — хуками, а не инструкцией

Инструкция в markdown работает вероятностно: модель может её не прочитать,
не так понять, или решить, что случай особый. Хук выполняется всегда и
детерминированно.

Правило: **всё, что должно быть верно всегда, — хук.** Всё, что требует
суждения, — инструкция. Коммит в `main` не требует суждения, значит хук.
«Не расширяй скоуп» требует, значит инструкция.

## Инициализация проекта

Плагин не может разворачивать файлы сам — платформа такого механизма не даёт
(см. ограничения в `CLAUDE.md`). Поэтому скаффолдинг реализован скиллом
`new-project`, который держит шаблоны в своём тексте и пишет их через Write.

Разделение получается вынужденным, но правильным:

- **плагин везёт поведение** — обновляется централизованно;
- **init везёт файлы** — материализуется в проекте, дальше живёт своей жизнью
  и может быть подстроен под конкретику.

Подробности — [init-procedure.md](init-procedure.md).

## Что осознанно НЕ делается

- **Не абстрагируем то, что встретилось один раз.** Абстракция на одном примере
  почти всегда абстрагирует не то. Ждём второго случая.
- **Не тащим зависимости.** Скрипты хуков — Node без `node_modules`. Харнес не
  должен требовать установки чего-либо в целевом проекте.
- **Не делаем плагин умным.** Плагин — набор текстов и мелких скриптов. Вся
  сложность вынесена в решения, а не в код.
- **Не пытаемся покрыть все стеки.** `harness.json` описывает команды строкой —
  этого достаточно для любого стека без поддержки его в коде харнеса.
