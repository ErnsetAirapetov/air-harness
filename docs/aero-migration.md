# Миграция AERO на плагин harness-core

Одноразовая инструкция для владельца. Применяется в сессии AERO
(github.com/ErnsetAirapetov/AERO, Go, дефолтная ветка `dev`). Смысл: доска
переезжает из self-hosted Plane в GitHub issues, гейты и роли — в плагин; из
`.claude/` проекта уходит всё, что теперь приходит из харнеса. После живого
прогона (фаза 6) этот документ можно удалить.

Каждая фаза — чек-лист. Порядок жёсткий: **демонтаж Plane-обвеса (фаза 4) — только
после сверки переноса (фаза 3).** Данные Plane до сверки не трогаем.

## Решения владельца (2026-07-21) — обязательны к исполнению

- **Переносим ВСЕ задачи Plane, включая «Готово».** Готовые становятся
  закрытыми issues — полная история в одном месте.
- **Plane после сверки замораживается read-only архивом, не гасится.**
- **Главная ветка харнеса в AERO — `dev`** (как сейчас). Конвенция веток
  `feature/<slug>` сохраняется — под трекер не переименовываем.
- **Приоритет Plane переносим метками `priority:*`** (сохранение исторического
  поля). Оговорка: харнес-конвенция приоритет метками не выражает — для новых
  задач приоритет живёт в порядке бэклога (см. `docs/task-hierarchy.md`).

## Безопасность (публичный репозиторий харнеса)

Ни в этот документ, ни в тело GitHub-issues **не выносить** инфраструктуру Plane
дословно: хост доски, UUID проекта/статусов/модулей, email/UUID сервисного
аккаунта, содержимое `settings.local.json`. Все константы доступа (эндпоинт REST,
ключ, UUID) агент берёт из локальных `.claude/skills/plane-task/SKILL.md` и
`.claude/agents/plane-scribe.md` сессии AERO — они там и остаются до заморозки.
AERO-репозиторий **публичный**: если светить хост Plane в issues нежелательно,
обратной ссылкой ставь идентификатор `AERO-<N>` без полного URL (см. фазу 3).

## Фаза 1. Уборка перед стартом

Сейчас `dev` ahead 3 от `origin/dev`, в рабочем дереве — незакоммиченные
изменения worktree-артефактов (`.claude/worktrees/`). До миграции:

1. Разобрать незакоммиченное на `dev`: доделать и закоммитить по правилам AERO
   или отложить (`git stash`). Гейт `bash scripts/verify.sh` зелёный.
2. `git push` — `origin/dev` синхронизирован.
3. Прибрать залежавшиеся worktree: `git worktree list` → для каждого влитого
   `git worktree remove <путь>` + удалить локальную ветку. Каталоги
   `.claude/worktrees/{aero-autoclose, aero70-empty-work, m4-knowledge-base,
   m3e-hardening}` — проверить, что за ними нет незалитой работы, и убрать.
4. `git status` чист, `git worktree list` — только основной чекаут.

## Фаза 2. `/harness-init` в сессии AERO

Запусти `/harness-init`. Он сам создаёт привязки, метки, шаблон задачи, защиту
ветки, `orchestrator.md` и мержит `settings.json` — шаги не пересказываем.
Задай на его вопросы вход:

- **Главная ветка:** `dev` (init предложит её из `origin/HEAD` — подтвердить).
- **Фордж:** GitHub (`gh`) — определится из remote сам.
- **Гейт проверок (`checks`):** существующий `bash scripts/verify.sh`
  (go build + go vet + go test с гейтом покрытия `-coverpkg=./...`, порог
  `COVERAGE_MIN`). `editChecks` — быстрая на `.go`:
  `{ "match": "\\.go$", "run": "go vet ./..." }`.
- **Области (`area:*`):** перечисли текущий канон AERO из `plane-task`
  (orchestrator, engine, claude, telegram, beads, director, vcs, library,
  prompts, config, deploy, infra, product) — миграция требует их 1:1.
  Укрупнение до 3–6 областей — отдельный груминг после, не сейчас.
- **Оркестратор:** имя по предложению init. **Строгость:** `strict: true`.

Ожидаемый результат: `.claude/harness.json` (mainBranch `dev`), `.claude/forge.md`
(GitHub), `settings.json` смержен с подключением плагина, метки харнес-осей и
шаблон задачи заведены, защита ветки `dev` на GitHub включена. Существующие
`CLAUDE.md`/`AGENTS.md`/`docs/` scaffolder **не перезаписывает** — их правим
вручную в фазе 4. Плагин мог быть подключён на ходу — если хартии оркестратора
в контексте нет, это не сбой (лечится `/compact`; см. README, «Если что-то не
работает»).

## Фаза 3. Перенос Plane → GitHub

Механическая выгрузка-заливка — рецепт агенту-исполнителю в сессии AERO
(константы доступа он берёт из локального `plane-task`/`plane-scribe`; на
Windows кириллицу шли `ensure_ascii=False`, вывод — `stdout.reconfigure("utf-8")`,
иначе консоль падает на cp1251). Порядок — милстоуны раньше issues, чтобы
привязки сходились.

**Шаг A. Выгрузить Plane списком (REST).** MCP на self-hosted CE отдаёт 404 на
`list_modules` — только REST. Выгрузи в JSON: модули (`имя → id`) и все
work-items с полями `sequence_id, name, description_html, state, labels,
priority, module, parent, links`. Зафиксируй счётчики: сколько задач в
«Готово» / «В работе» / «Бэклог» — понадобятся для сверки.

**Шаг B. Милстоуны из модулей.** Каждый модуль Plane (`Эпик N — <название>
(этап <код>)`) → милстоун GitHub с тем же именем:
`gh api repos/{owner}/{repo}/milestones -f title="Эпик N — …"`. Сохрани
`имя → number`. (Согласование этих имён с харнес-кодами `M1-E2` —
позднейший груминг, не эта миграция.)

**Шаг C. Метки.** Заведи недостающие (init создал только харнес-канон),
маппинг 1:1:

| Plane | GitHub |
|---|---|
| тип: `feat` `fix` `docs` `refactor` `test` `chore` | `type:feat` … `type:chore` |
| область: 13 из `plane-task` | `area:<то же>` |
| спец: `captured` `tech-debt` `research` `blocked` | как есть (`blocked` уже в каноне) |
| поле «Приоритет»: `urgent` `high` `medium` `low` | `priority:urgent` … `priority:low` |

**Шаг D. Issues по каждой задаче** (пакетно `gh issue create` / `gh api`):

- Заголовок = имя задачи Plane (по-русски).
- Тело = описание Plane (HTML `<p>/<a>/<b>` GitHub рендерит) + обратная ссылка
  `перенесено из Plane: AERO-<sequence_id>` (URL — по решению из блока
  «Безопасность»). Ссылки на MR/спеки из Plane Links — тем же телом.
- Метки: тип + область + спец + `priority:*` по маппингу шага C.
- Милстоун: `--milestone "Эпик N — …"` по модулю задачи.
- **Статус:** «Готово» → создать и сразу `gh issue close <N>`; «В работе» /
  «Бэклог» → оставить открытым.

**Шаг E. Контрольная сверка (обязательна до фазы 4).** Количество issues по
состояниям должно сойтись с счётчиками шага A:
`gh issue list --state closed --limit 1000 --json number | jq length` = число
«Готово»; открытых = «В работе» + «Бэклог». Разошлось — ищи потерянные задачи,
демонтаж не начинать.

## Фаза 4. Демонтаж старого обвеса (после сверки фазы 3)

Удалить (заменено плагином/переехало):

| Что удалить | Чем заменено |
|---|---|
| `.claude/hooks/no-ai-authorship.sh` | git-гейт харнеса `guard-git` (блокирует Co-Authored-By, упоминания ассистента, `--author` не владельца, коммит/пуш в главную) |
| секция `hooks` в `.claude/settings.json` (регистрирует тот хук) | хуки приходят из плагина |

| `.claude/agents/plane-scribe.md` | роли исполнителей — из плагина (`impl`, `reviewer`, `groomer`) |
| сервер `plane` в `.claude/.mcp.json` | Plane больше не пишем из сессии |

Вычистить упоминания Plane и старого хука в доках проекта (конкретные места):

- **`CLAUDE.md`** — строка про «блокируется хуком
  `.claude/hooks/no-ai-authorship.sh`»: заменить на «git-гейтом харнеса».
- **`AGENTS.md`** — §1 «трекер — Plane (§10)» → «трекер — GitHub issues
  (`.claude/forge.md`)»; §3 строка «механика Plane — haiku (`plane-scribe`)» →
  убрать (роли из плагина); §7 «Enforced хуком `.claude/hooks/no-ai-authorship.sh`;
  … четырьмя слоями (D20)» → «git-гейт харнеса (guard-git)»; **весь §10 «Задачи
  в Plane»** — заменить коротким разделом про GitHub issues со ссылкой на
  `forge.md` и `task-hierarchy.md`, либо удалить.
- **`docs/DECISIONS.md`** — сверить ADR **D20** (четыре слоя запрета
  ИИ-авторства): локальный хук-слой заменён гейтом плагина — обновить статус/текст
  тем же коммитом (дисциплина решений AERO §2).

Всё — обычным коммитом на ветке `feature/<slug>` от `dev` через PR (не на `dev`
напрямую — гейт заблокирует).

## Фаза 5. Заморозка Plane

- Пометить проект/доску Plane архивом: описание доски — «архив, живая доска —
  GitHub issues репозитория AERO». Задачи и статусы не менять — read-only ссылка.
- Проект **не гасить** (решение владельца) — остаётся историей.
- После заморозки убрать ключ `PLANE_API_KEY` из `.claude/settings.local.json`
  (в git его нет и не было).

## Фаза 6. Смоук-чек харнеса (принцип живого прогона)

1. Плагин виден: `/agents` и `/plugin` показывают роли и скиллы харнеса. Не
   видно — разбираться с установкой (README, «Если что-то не работает»).
2. Хартия оркестратора: после перезапуска/`/compact` оркестратор представляется
   заданным именем и делегирует исполнение, а не пишет код сам.
3. Гейт: попроси агента `git commit` на `dev` — должен быть заблокирован с
   внятной причиной (guard-git); то же для коммита с `Co-Authored-By`.
4. `/board` — сводка собирается, милстоуны-эпики и `area:*` группируются.
5. `/task <мелкая задача>` — агент в worktree, PR открыт, `Closes #N` на месте,
   `bash scripts/verify.sh` из `checks` реально гоняется (красный не даст
   закончить молча).
6. `/land <PR>` — ревью, мерж, worktree убран, задача закрыта.

Что-то ведёт себя не так — это дефект харнеса: чинить в `air-harness` (issue
этого репозитория), не локальными заплатками в AERO.
