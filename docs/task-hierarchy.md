# Иерархия задач

Единая схема разбивки работ для всех проектов харнеса. Термины и коды отсюда
обязательны: их используют доска, roadmap, имена веток, groomer при
декомпозиции и владелец в разговоре с оркестратором.

## Два измерения — не путать

| Измерение | Природа | Носитель |
|---|---|---|
| **Иерархия**: Веха → Эпик → Задача | конечное, закрывается | roadmap + доска |
| **Направление**: `area:*` | постоянное, не закрывается | метка на задаче |

«Закончить netcode» не бывает — это направление. «Закончить веху M2» —
бывает: у вехи есть проверяемое состояние готовности. Направления
(combat, netcode, infra…) режут кодовую базу, вехи режут время.

## Три уровня

| Уровень | Код | Носитель | Размер | Что это |
|---|---|---|---|---|
| Веха | `M1` | `docs/roadmap.md` + милстоун форджа | 2–6 эпиков | проверяемое состояние продукта («играбельный прототип») |
| Эпик | `M1-E2` | tracking-задача с чеклистом | 3–12 задач | связный блок функциональности |
| Задача | `M1-E2-T3` | обычная задача на доске | 1–3 файла, до ~300 строк диффа | один заход одного агента: 1 задача = 1 ветка = 1 PR |

Четвёртого уровня нет и не будет. Эпик перерос 12 задач — это два эпика,
а не повод для нового уровня вложенности.

## Коды

- Титул задачи на доске: `[M1-E2-T3] Нормализация точек`.
- Титул tracking-задачи эпика: `[M1-E2] Распознавание жестов`,
  в теле — чеклист `- [ ] #NN` по номерам задач доски.
- Ветка: `<type>/m1-e2-t3-<слаг>` (код в нижнем регистре).
- Номера уникальны в пределах родителя и никогда не переиспользуются:
  закрытый `M1-E2` навсегда остаётся `M1-E2`.
- В `docs/roadmap.md` живут только вехи и эпики. Задачи живут на доске —
  дублировать их в roadmap запрещено (один факт — одно место).

## Внеплановая работа

Не всё обязано жить в иерархии. Багфиксы, мелкие chore и внезапные
исследования заводятся задачами **без кода** — только метки `type:fix` /
`type:chore` / `type:research` и `area:*`. Правило простое: код получает
лишь то, что двигает какую-то веху. Если «внеплановых» задач становится
больше трети доски — это сигнал, что roadmap разошёлся с реальностью,
и нужен груминг.

## Метки задачи

Ровно по одной с каждой оси: `area:*`, `type:*` (design / feat / fix /
chore / research), `size:*` (S / M / L). Плюс потоковые `needs-decision`,
`blocked`. Приоритет метками не выражается — только порядком в бэклоге.

## Правила движения

- Веха закрыта = все её эпики закрыты и заявленное состояние продукта
  фактически проверено (не «код написан», а «в это можно поиграть»).
- Эпик закрыт = все задачи чеклиста закрыты.
- Задача не влезает в один PR → это не задача, а эпик: вернуть в груминг.
- Открытых вех не больше двух: текущая и, ближе к её концу, следующая.
  Вехи «на будущее» в roadmap — списком без эпиков, одной строкой каждая.
- Задача с двумя `area:*` — почти всегда две задачи. Дели.

## Кто это применяет

- **groomer** — при декомпозиции присваивает коды, собирает tracking-задачи.
- **/board** — группирует бэклог по вехам и area.
- **scaffolder** — закладывает `docs/roadmap.md` с вехой M1 при инициализации.
- **impl** — берёт код задачи в имя ветки.
