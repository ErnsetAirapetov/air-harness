# Состояние работ

Документ для того, кто продолжает работу над харнесом. Обновляется по мере
готовности компонентов.

## Как проверять

```bash
node tests/guard-git.test.mjs      # гейт git-процесса
node tests/run-checks.test.mjs     # гейт проверок
node tests/edit-checks.test.mjs    # проверки после правки
claude plugin validate ./core      # манифест плагина
```

Предупреждение валидатора об отсутствии `version` — ожидаемое и намеренное,
см. раздел «Версионирование» в `CLAUDE.md`.

Тесты поднимают временный репозиторий с главной веткой `trunk`, а не `main` —
так проверяется, что скрипты берут имя ветки из конфига проекта, а не знают
его сами. Приём сохранять во всех будущих тестах гейтов.

Вход хука в тестах собирается в Node и передаётся через stdin. Попытка
построить JSON в shell ломается на экранировании и даёт **ложно-зелёный**
результат, когда скрипт молча выходит с нулём на неразобранном входе.
На эти грабли уже наступали.

1. **Живой прогон на ink-arena** — владелец применяет
   [ink-arena-migration.md](ink-arena-migration.md) и проходит чек-лист оттуда.
   Ничто из «готово» выше не считается доказанным до этого прогона (принцип 8).

2. **Гейты на фордже** — настройка серверной защиты главной ветки и
   merge-чеков добавлена в `harness-init` (подраздел «Гейты на фордже»);

3. **По результатам прогона** — решить, нужен ли отдельный скилл `new-project`
   или ветвления внутри `harness-init` достаточно.

## Чего делать НЕ надо

- Не добавлять роли и команды «на будущее» — только под реальную потребность
  (принцип 3). Отдельного `doc-writer` в плагине нет намеренно: документные
  роли предметны, живут в проектах (пример — `gdd-writer` в ink-arena).
- Не тащить зависимости: скрипты хуков остаются на голом Node.
- Не переносить в `core/` ничего, где упоминается конкретный фордж, стек,
  имя ветки или email. Это ошибка слоя, см. `CLAUDE.md`.
- Не раздувать тексты: агент до ~60 строк, команда до ~30 (принцип 7).

## Известные вопросы

- `run-checks.mjs` доверяет `cwd` из входа хука как каталогу, где работал
  агент (в проектной версии ink-arena это подтверждено вживую). Проверить то
  же для плагинной подписки на `SubagentStop` при живом прогоне.
- Формат `enabledPlugins` / `extraKnownMarketplaces` подтверждён живой
  установкой (объект `{"plugin@marketplace": true}`; в маркетплейсе работает
  `"autoUpdate": true`).
- Если на `Stop` и `SubagentStop` проверки будут гоняться дважды за один цикл —
  оставить только `SubagentStop`.
- Проверить вживую SessionStart-инъекцию: stdout хука попадает в контекст на
  всех трёх matcher-ах (startup / clear / compact), хартия не дублируется,
  её токен-стоимость терпима (~250 токенов на сессию).
- Автоперезапуск по 5-часовому лимиту — полуручной (планировщик ОС по
  `/handoff limit`): изнутри сессии остаток лимита не виден, лучшего
  механизма у платформы нет. Пересмотреть, когда появится.
- Проверить при живом прогоне зависимость `superpowers`: автоустановку при
  установке harness-core через маркетплейс и доступность скиллов
  `superpowers:*` внутри сабагента `impl` (у него полный набор инструментов,
  включая Skill; у reviewer/groomer инструменты урезаны — им скиллы и не
  нужны, кроме brainstorming, который вызывает координатор, а не groomer).
