готовности компонентов.

## Как проверять

```bash
node tests/guard-git.test.mjs      # гейт git-процесса
claude plugin validate ./core      # манифест плагина
```

Предупреждение валидатора об отсутствии `version` — ожидаемое и намеренное,
см. раздел «Версионирование» в `CLAUDE.md`.

Тест поднимает временный репозиторий с главной веткой `trunk`, а не `main` —
так проверяется, что скрипт берёт имя ветки из конфига проекта, а не знает
его сам. Этот приём стоит сохранять во всех будущих тестах гейтов.

Важно про построение тестов: вход хука собирается в Node и передаётся через
stdin. Попытка построить JSON в shell ломается на экранировании кавычек и
обратных слэшей — и даёт **ложно-зелёный** результат, когда скрипт молча
выходит с нулём на неразобранном входе. На эти грабли уже наступали.

## Что дальше

По убыванию важности.

### 1. Остальные роли — `core/agents/`

По образцу `impl.md`: фордж-независимые, читают `.claude/forge.md`.

- `reviewer` — ревьюер PR перед мержем. Только чтение. Проверяет критерии
  приёмки построчно, соблюдение Границ, соглашения проекта, реальные дефекты.
  Возвращает вердикт МЁРЖИТЬ / ДОРАБОТАТЬ / ПЕРЕДЕЛАТЬ и, если ДОРАБОТАТЬ, —
  готовый промпт для `impl`.
- `groomer` — декомпозиция цели в атомарные задачи по шаблону. Одна задача =
  одна ветка = один PR = один заход агента, 1–3 файла, до ~300 строк диффа.
- `doc-writer` — документация. Модель `sonnet`.

### 2. Ритуалы — `core/commands/`

- `/board` — сводка доски для человека: что ждёт решения, что в работе, что
  готово к мержу, аномалии (worktree без PR, задачи без меток, зависшие ветки).
- `/task <N...>` — взять задачи в работу, спаунить агентов параллельно одним
  сообщением, с `isolation: "worktree"`.
- `/land <PR...>` — ревью через `reviewer`, по вердикту мерж или доработка,
  затем обязательная уборка worktree и закрытие задачи.
- `/groom <цель>` — декомпозиция; **показать таблицу задач человеку до
  создания** и дождаться подтверждения.

Команды тоже не должны знать про `gh`. Всё — через `.claude/forge.md`.

### 3. Инициализация — `core/skills/`

- `new-project` — пустой репозиторий с нуля.
- `harness-init` — добавление харнеса в живой репозиторий.

Спецификация целиком в [init-procedure.md](init-procedure.md) — реализовывать
по ней. Ключевое: скилл держит шаблоны файлов **в своём тексте** и пишет их
через Write, потому что `${CLAUDE_PLUGIN_ROOT}` в теле скиллов не
подставляется, а механизма шаблонов у платформы нет.

Внутри должны быть готовые заготовки `forge.md` минимум для GitHub (`gh`) и
GitLab (`glab`) — это главная ценность процедуры.

### 4. Проверка на живом проекте

Ничего из перечисленного не считается готовым, пока не прогнано через
`claude --plugin-dir <путь>/core` в реальном репозитории. Подопытный —
`ink-arena`: там уже есть работающая проектная версия этих же компонентов,
с которой можно сверяться.

После проверки — перевести `ink-arena` на плагин: убрать из его `.claude/`
всё, что теперь приходит из харнеса, оставить только привязки и предметный
слой. Это будет первым настоящим доказательством, что расслоение верное.

## Чего делать НЕ надо

- Не добавлять роли и команды «на будущее» — только под реальную потребность
  (принцип 3).
- Не тащить зависимости: скрипты хуков остаются на голом Node.
- Не переносить в `core/` ничего, где упоминается конкретный фордж, стек,
  имя ветки или email. Это ошибка слоя, см. `CLAUDE.md`.
- Не раздувать тексты: агент до ~60 строк, команда до ~30 (принцип 7).
