# Процедура инициализации проекта

Спецификация того, что делают скиллы `new-project` (пустой репозиторий) и
`harness-init` (репозиторий, где уже есть код). Это документ-источник: скиллы
реализуют написанное здесь, а не наоборот.

Цель — заменить час ручной возни при заведении каждого нового репозитория,
микросервиса или прототипа на один вызов команды.

## Принципы процедуры

- **Спрашивать мало.** Всё, что выводится из репозитория, выводится, а не
  спрашивается. Максимум 4 вопроса человеку.
- **Ничего не затирать.** Существующие файлы не перезаписываются без явного
  подтверждения. `harness-init` в живом репо особенно осторожен.
- **Идемпотентность.** Повторный запуск не ломает уже настроенное, а
  дополняет недостающее.
- **Результат проверяем.** В конце — самопроверка и отчёт, что реально создано.

## Шаг 0. Разведка (без вопросов)

Определить самостоятельно:

| Что | Как |
|---|---|
| Фордж | `git remote -v` → github.com / gitlab.* / прочее |
| Стек | наличие `package.json`, `pyproject.toml`, `go.mod`, `Cargo.toml`, `*.csproj` |
| Команды проверок | скрипты из манифеста стека (`test`, `lint`, `typecheck`) |
| Главная ветка | `git symbolic-ref refs/remotes/origin/HEAD`, иначе текущая |
| Email владельца | `git config user.email` |
| Пустой ли репо | `git log` пуст → сценарий `new-project` |

Всё найденное показать человеку одним блоком как «вот что я понял» — и
спрашивать только то, что не определилось или где есть выбор.

## Шаг 1. Вопросы человеку (не более четырёх)

1. **Назначение проекта** — 1–3 предложения. Идёт в `CLAUDE.md` и `README.md`.
   Единственный по-настоящему обязательный вопрос.
2. **Эпики** — крупные направления работ, 3–6 штук. Станут метками `epic:*`.
   Предложить варианты, исходя из назначения.
3. **Стек** — только если разведка не определила.
4. **Строгость гейтов** — блокировать ли коммит при красных проверках, или
   только предупреждать.

## Шаг 2. Что создаётся

### `.claude/harness.json` — привязки для скриптов

```json
{
  "mainBranch": "main",
  "ownerEmail": "eriktarakan@gmail.com",
  "worktreeDir": ".claude/worktrees",
  "checks": ["npm test", "npx tsc --noEmit"],
  "strict": true
}
```

### `.claude/forge.md` — таблица перевода к доске задач

Заполняется по выбранному форджу. Формат — в
[architecture.md](architecture.md#фордж--claudeforgemd).

### `.claude/settings.json` — permissions и подключение плагина

Permissions плагин поставлять не может, поэтому генерируются здесь: разрешить
CLI форджа, менеджер пакетов, безопасные подкоманды git; запретить чтение
`.env`, ключей и сертификатов.

Туда же — `extraKnownMarketplaces` и `enabledPlugins`, чтобы харнес
подхватывался у любого, кто склонирует репозиторий.

### `CLAUDE.md` проекта

Только предметный слой: назначение, стек, команды запуска, соглашения домена.
**Процесс не дублируется** — он приходит из плагина. Ссылка на `.claude/forge.md`
и `.claude/harness.json`.

Если `CLAUDE.md` уже есть — не перезаписывать, а предложить дополнение
отдельным блоком.

### `.gitignore`

Дописать `.claude/worktrees/` и `.claude/settings.local.json`, если их нет.

### Доска задач

- Метки: `epic:*` по ответу человека, `type:*` (design/feat/fix/chore/research),
  `size:*` (S/M/L), потоковые `needs-decision` и `blocked`.
- Удалить дефолтные шаблонные метки форджа (`bug`, `enhancement`, `question`…) —
  они дублируют оси и создают шум.
- Шаблон задачи: `.github/ISSUE_TEMPLATE/task.yml` для GitHub,
  `.gitlab/issue_templates/task.md` для GitLab. Обязательные разделы: Контекст,
  Задача, Критерии приёмки, Границы, Модель.

### Первые задачи

Завести 3–5 стартовых issue, выведенных из назначения проекта, чтобы доска не
была пустой и первый `/task` можно было запустить сразу.

### `docs/`

`docs/README.md` — карта документации. Для нового проекта также заготовки
`docs/vision.md` и `docs/decisions.md` (журнал решений).

## Шаг 3. Самопроверка

Проверить и доложить фактическое состояние, а не намерения:

- `.claude/harness.json` — валидный JSON, `checks` реально выполняются;
- метки заведены (перечитать с форджа, а не поверить своей же команде);
- шаблон задачи на месте;
- `git status` — показать человеку, что появилось;
- хуки харнеса: прогнать `guard-git.mjs` с тестовым входом, убедиться, что он
  блокирует коммит в главную ветку и пропускает коммит в ветке задачи.

## Шаг 4. Отчёт

Короткая сводка: что создано, какие метки заведены, какие задачи стоят в
бэклоге, что делать дальше (`/board`, затем `/task N`). Плюс явный список
того, что **не** удалось, если что-то не удалось.

## Отличия `harness-init` для живого репозитория

- Ничего не перезаписывать; про каждый конфликт спрашивать отдельно.
- `CLAUDE.md` — если есть, дополнить, а не заменить.
- Существующие метки не удалять вслепую: показать, что предлагается убрать.
- Существующие задачи не трогать, метки к ним не приписывать.
- Дополнительно: разведать текущие соглашения репозитория (стиль коммитов по
  `git log`, структура каталогов) и отразить их в `CLAUDE.md`, а не навязывать
  свои.
