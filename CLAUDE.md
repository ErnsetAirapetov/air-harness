# CLAUDE.md — air-harness

Это репозиторий **харнеса**, а не обычного проекта. Здесь лежит инструментарий,
который затем подключается плагином в другие репозитории и управляет работой
агентов там. Отсюда все особенности ниже.

## Главное отличие от обычного репо

Ошибка в коде проекта ломает один проект. **Ошибка здесь ломает все проекты
сразу** — и, что хуже, ломает их тихо: агент просто начинает вести себя иначе,
а вы узнаёте об этом через неделю по кривому PR. Поэтому здесь:

- изменения делаются мелкими и проверяются на живом проекте до мержа;
- любое правило имеет за спиной конкретный случай, когда без него было плохо;
- умозрительных улучшений «на будущее» не вносим.

## Что где лежит

```
.claude-plugin/marketplace.json   каталог плагинов этого репо
core/                             плагин harness-core
  .claude-plugin/plugin.json      манифест
  agents/                         роли исполнителей
  commands/                       ритуалы человека
  skills/                         процедуры (в т.ч. инициализация проектов)
  hooks/hooks.json                подключение гейтов
  scripts/                        код гейтов (Node, без зависимостей)
docs/                             архитектура, принципы, статус
```

## Границы слоёв — соблюдать строго

Три слоя, и путать их нельзя. Это главный инвариант репозитория.

| Слой | Где живёт | Что туда можно |
|---|---|---|
| Процесс | плагин `core/` | Всё, что верно для **любого** проекта на **любом** стеке и **любом** фордже |
| Привязки | генерируется в целевой проект | Команды форджа, команды стека, имя главной ветки, email владельца |
| Проект | `CLAUDE.md` целевого проекта | Домен, модули, соглашения конкретного проекта |

Практическое правило: **если в файле внутри `core/` встречается `gh`, `npm`,
`pytest`, `main` или конкретный email — это ошибка слоя.** Такие вещи читаются
из `.claude/harness.json` и `.claude/forge.md` целевого проекта.

Исключение: примеры внутри документации и шаблоны, которые init-скилл пишет в
целевой проект, — там конкретика уместна, она там и должна материализоваться.

Исключение: `gh` и адрес `ErnsetAirapetov/air-harness` в скилле
`harness-feedback` — не ошибка слоя. Канал фидбэка ведёт в собственный дом
харнеса, который константен для всех проектов и не зависит от форджа целевого
проекта, — поэтому и адрес, и команда захардкожены намеренно, а не читаются из
привязок.

## Ограничения платформы — знать до того, как проектировать

Проверено по докам, обходить бесполезно:

- Плагин **не может** поставлять `CLAUDE.md` — он не грузится как контекст.
  Правила несут агенты и скиллы с внятным `description`. Канал в главную
  сессию — SessionStart-хук: его stdout попадает в контекст (так
  доставляется хартия оркестратора).
- Плагин **не может** поставлять `permissions`. В его `settings.json` работают
  только `agent` и `subagentStatusLine`. Permissions генерирует init-скилл в
  `.claude/settings.json` целевого проекта.
- Механизма шаблонов файлов нет. Скаффолдинг = скилл, который держит шаблоны в
  своём тексте и пишет их через Write.
- `${CLAUDE_PLUGIN_ROOT}` подставляется в командах хуков, MCP и LSP, но **не** в
  теле скиллов и агентов. Скилл не может сослаться на файл внутри плагина по
  пути — значит всё нужное несёт в себе.
- `agents/`, `commands/`, `skills/` и `hooks/hooks.json` подхватываются
  автоматически по именам. Указывать `hooks` в `plugin.json` при этом
  **нельзя** — тот же файл загрузится дважды, и установка падает с ошибкой
  «Duplicate hooks file detected» (проверено на живой установке). Поле
  `hooks` — только для дополнительных файлов хуков.

## Версионирование

Поле `version` в `plugin.json` намеренно **отсутствует**. Тогда версией
становится SHA коммита. Это осознанный выбор: единственный потребитель —
владелец, а цель — «починил в одном месте, работает везде».

Оговорка: для сторонних маркетплейсов автообновление по умолчанию выключено —
на каждой машине его нужно один раз включить (см. README, раздел
«Автообновление»). С включённым автообновлением обновления подтягиваются
после старта сессии.

Следствие: **пуш в `main` меняет поведение агентов во всех проектах со
следующей сессии.** Отсюда требование ниже.

## Процесс работы в этом репозитории

- Главная ветка — `main`, напрямую не коммитим. Только через PR.
- Каждая задача — своя ветка и свой git worktree в `.claude/worktrees/`.
- Изменения в `core/` перед мержем проверяются на живом проекте:
  `claude --plugin-dir <путь>/core` в реальном репозитории, прогон затронутого
  ритуала. «Выглядит правильно» — не проверка.
- Манифесты валидируются: `claude plugin validate ./core`.
- Хуки — это код, который может заблокировать работу во всех проектах. Правки
  в `core/scripts/` тестируются прогоном скрипта с примером входного JSON.

## Соглашения

- Conventional Commits, сообщения на русском, кратко но по существу.
- Автор коммитов только владелец (`eriktarakan@gmail.com`). Никаких упоминаний
  ассистентов в авторстве и сообщениях, никаких trailer-ов `Co-Authored-By`.
- Скрипты хуков — Node без внешних зависимостей. Харнес не должен требовать
  установки чего-либо в целевом проекте.
- Документация — в `docs/`, на русском.
- Изменил состав `commands/`, `skills/`, `agents/` или хуков — обнови каталог
  возможностей в README тем же PR.
- Тексты агентов и команд пишутся так, будто их прочитает агент, у которого нет
  никакого контекста кроме этого файла. Потому что так и есть.
